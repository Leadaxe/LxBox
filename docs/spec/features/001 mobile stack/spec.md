## 1. Зачем отдельный документ о стеке

BoxVPN — клиент для **Android и iOS** с общим кодом интерфейса и сценариев, работающий поверх **sing-box** как встроенного ядра. Стек выбран так, чтобы:

- поддерживать **две мобильные ОС** без дублирования экранов и навигации;
- соблюдать **требования платформ** к VPN (отдельные API и процессы/расширения);
- иметь предсказуемый путь интеграции **libbox** (официальная модель встраивания sing-box на мобильных устройствах).

Ниже — состав стека и причины выбора каждого слоя.

## 2. Слой UI и прикладной логики: Flutter и Dart

**Используем:** **Flutter** (стабильная ветка 3.x) и язык **Dart**.

**Почему один фреймворк на обе платформы**

- Flutter рисует UI собственным движком (**Skia / Impeller**), а не переводит виджеты в нативные контролы каждой ОС. В результате **один и тот же код** описывает вёрстку и поведение экранов на Android и iOS, с предсказуемым видом и меньшим числом «разъехалось на iOS» из-за различий UI-kit’ов.
- Для мобильного клиента с формами, списками профилей, настройками и статусом подключения это снижает стоимость сопровождения по сравнению с **двумя независимыми UI** (отдельно Jetpack Compose и SwiftUI).

**Почему Dart**

- Dart — язык по умолчанию для Flutter: **одна модель асинхронности** (`async`/`await`, `Future`, `Stream`), строгая типизация, один набор линтеров и инструментов для обеих целей сборки.

**Интеграция с низким уровнем**

- У Flutter есть **platform channels** для вызова **Kotlin** и **Swift** из Dart с передачей примитивов и бинарных буферов — этого достаточно для команд «запусти туннель», «останови», «отдай статус».
- При необходимости прямого вызова C/Go API существует **`dart:ffi`**; выбор между channel и FFI фиксируется в техспеках по модулям, здесь важно, что оба механизма **поддержаны** и не требуют смены фреймворка.

**Почему не другие варианты для общего UI** (кратко)

| Подход | Почему не основной выбор |
|--------|---------------------------|
| **Два нативных UI** (Compose + SwiftUI) | Две кодовые базы экранов, двойные правки фич и регрессий. |
| **Kotlin Multiplatform + Compose Multiplatform** | Разумный запасной путь, если команда целиком на Kotlin; общий UI на iOS у стека исторически развивался иначе, чем у Flutter, объём компромиссов по платформам нужно оценивать отдельно. |
| **React Native** | Для сценариев с постоянным **FFI**, фоновыми процессами и политиками магазинов по VPN чаще больше трения, чем у Flutter + нативного слоя. |

Итог: **Flutter + Dart** — основной способ получить **одну кодовую базу UI** для Android и iOS при сохранении нативного VPN-слоя ниже.

## 3. Ядро: sing-box как библиотека (libbox)

**Используем:** встроенное ядро **sing-box** в виде **libbox** (или эквивалентная официальная поставка для встраивания): сборки под **Android (JNI)** и **iOS (XCFramework)**.

**Почему встроенная библиотека, а не отдельный исполняемый файл**

- На мобильных ОС пользовательский трафик в типичной схеме идёт через **VPN API** и процесс/расширение, которое держит туннель. Ожидаемая модель — **один процесс приложения + нативная библиотека**, а не запуск произвольного бинарника из файловой системы (политика безопасности и жизненный цикл приложения).

**Версионирование**

- Версию libbox **жёстко привязываем** к релизу приложения (воспроизводимые сборки, одинаковое поведение у всех пользователей версии X).
- **Автообновление ядра по сети** из приложения — отдельное продуктовое решение; в этом документе не фиксируется.

## 4. Платформенный слой VPN (нативный код, не Flutter)

ОС требуют, чтобы перехват трафика шёл через **свои API**:

- **Android:** подкласс **`VpnService`**, при необходимости **foreground service** и постоянное **уведомление**; из **Kotlin** вызывается libbox (JNI или сгенерированные биндинги).
- **iOS:** расширение типа **Network Extension** (например, **Packet Tunnel**), контейнер приложения и мост к libbox на **Swift** (или Objective-C при необходимости).

**Почему этот слой не пишется на Dart**

- Flutter не заменяет системные контракты VPN: их нужно реализовать в **нативных** модулях, которые регистрируются в манифесте (Android) и в target расширения (Xcode).

Связь с Flutter — только через **platform channels** (и при необходимости **FFI**), без дублирования логики туннеля в Dart.

## 5. Границы ответственности

Зафиксированные правила, чтобы слои не протекали друг в друга:

| Слой | Ответственность | Чего не делает |
|------|-----------------|----------------|
| **UI** (Flutter) | Отображение состояния, ввод пользователя, навигация. | Не вызывает **API libbox** и не содержит вызовов JNI/FFI к ядру. Любое взаимодействие с туннелем — только через **абстрактный контракт** (сервис домена / тонкий platform bridge), выраженный в терминах команд и событий приложения, а не sing-box. |
| **Domain** (Dart) | Правила «когда можно подключаться», выбор профиля/конфига, агрегация статистики для UI, политика повторов и сообщений об ошибках **в продуктовых терминах**. | Не знает **деталей платформы** (классы `VpnService`, Network Extension, конкретные permission API). Видит только **порт** «туннель»: команды на уровне intent и события из модели **§6** в рамках **контракта bridge** (**§7**) без привязки к Android/iOS. |
| **Native** (Kotlin / Swift) | Регистрация VPN у ОС, жизненный цикл сервиса/расширения, вызов **адаптера libbox** (старт/стоп/загрузка конфига так, как требует встраивание), пересылка сырых колбэков ядра наружу как нормализованные события. | Не содержит **бизнес-логики** (выбор сервера по правилам продукта, подписки, парсинг URL провайдера) и **не строит UI**. Нативный код не решает «какой профиль активен» — только исполняет согласованную с доменом команду и сообщает результат. |

**Итог:** libbox сосредоточен за **нативным адаптером**; **UI** опирается на домен и на унифицированный канал событий; **домен** остаётся тестируемым и переносимым без знания Kotlin/Swift.

## 6. Потоки событий (event model)

Единая модель событий для связи нативного слоя и Flutter-домена (имена — стабильные идентификаторы для кода и доков):

| Событие | Эмитент (типично) | Потребители | Смысл |
|---------|-------------------|-------------|--------|
| **`connectRequested`** | UI → Domain инициирует; Domain подтверждает и передаёт в Native | Native готовит туннель | Пользователь или сценарий (автоподключение) запросил подключение; домен уже выбрал профиль/конфиг и передал **разрешённые** параметры нативному адаптеру. |
| **`tunnelStarted`** | Native (после успешного старта libbox + готовности туннеля у ОС) | Domain → UI | Туннель поднят, трафик может идти через sing-box согласно конфигу. |
| **`tunnelFailed`** | Native (ошибка libbox, ОС, таймаут, невалидный конфиг на стороне ядра) | Domain → UI | Подключение не состоялось; полезная нагрузка: **код/категория** для маппинга в текст и действия (повтор, открыть настройки). |
| **`statsUpdated`** | Native (периодический опрос или колбэки ядра) | Domain → UI | Обновление счётчиков/скорости для экрана статуса; частота оговаривается в реализации (throttle на Domain). |
| **`logsStreamed`** | Native (поток строк/чанков от libbox, если включено) | Domain (фильтр/уровень) → UI или буфер | Потоковые логи для отладки; в UI может быть отключён по умолчанию. |
| **`permissionRequired`** | Native (система отклонила VPN permission, нужен повторный запрос; специфика платформы) | Domain → UI | Нужно показать экран/диалог с объяснением и вызвать системный запрос разрешения; домен не вызывает API ОС напрямую — только отображает состояние и маршрутизирует пользователя. |

**Направление потоков (упрощённо):**

- Команды **вниз** (UI / Domain → Native): «подключи с этим конфигом», «отключи», «обнови лимиты логов» — без деталей libbox в UI.
- События **вверх** (Native → Domain → UI): строки таблицы выше; Domain переводит их в **состояние экрана** и тексты ошибок.

Детали сериализации (один channel vs несколько streams) — в техспеке модуля интеграции; имена событий остаются общими для Android и iOS.

## 7. Требования к Native Bridge Layer

Интеграция Flutter с нативным VPN-слоем требует **строгой дисциплины** проектирования bridge-слоя: ошибки на этой границе сильно усложняют отладку и сопровождение.

### 7.1 Требования

1. **Bridge layer минимален и тонок.** Не превращать его в «второй домен»: только то, что нужно для связи Dart ↔ Native и libbox-адаптера.
2. **Dart ↔ Native выполняет только маршаллинг и делегирование.** Сериализация/десериализация payload, вызовы в натив без **бизнес-логики** (ветвления по продуктовым правилам, выбор профиля и т.д. остаются в Domain).
3. **Публичный контракт API.** Все методы и каналы bridge, доступные с Dart-стороны, **фиксируются** как явный контракт (имена методов, типы аргументов, схемы сообщений). Любое изменение сигнатур или структур payload проходит **ревью как изменение публичного API** (версионирование при необходимости).
4. **Событийная модель формализована.** Статусы, ошибки, статистика, лог-стрим и события жизненного цикла туннеля описываются **единым event contract**, согласованным с перечнем **§6**; расширения перечисляются явно, а не добавляются «невидимыми» полями в произвольных map’ах.
5. **Логирование bridge в debug-сборках обязательно.** Диагностика вызовов и ответов на границе Dart/Native (с **редакцией** секретов: без полного конфига/ключей в логах по умолчанию).
6. **Нормализация ошибок.** Нативные ошибки приводятся к **единой модели** (код/категория/сообщение для UI), согласованной с Domain. **Запрещено** без обработки пробрасывать в Flutter platform-specific исключения, сырые stack traces ОС или нестабильные строки как единственный источник правды.

### 7.2 Обоснование

Без жёсткого контроля bridge быстро превращается в неструктурированный узел логики, что ухудшает:

- отладку интеграции Dart/Native/libbox;
- сопровождение **двух** платформ с дублированием «магии» в bridge;
- тестирование (контракт неясен — моки и регрессии дороже);
- замену или рефакторинг нативной реализации без ломки Flutter.
