# Tasks log

Журнал выполненных задач с отчётами — для истории и отчётности.

В отличие от `docs/spec/features/NNN/spec.md` (где описана фича как таковая — «что это и как работает»), tasks-файл описывает **конкретный рабочий цикл**: проблему, путь диагностики, решение, риски, критерии проверки.

Формат — один markdown-файл на задачу:

```
docs/spec/tasks/NNN-short-kebab-title.md
```

Номера монотонные, не переиспользуются. Если задача переросла в фичу — `spec.md` в `features/` появляется дополнительно, task остаётся как летопись.

## Секции в отчёте

```markdown
# NNN — Название задачи

| Поле | Значение |
|------|----------|
| Статус | Done / In progress / Blocked / Abandoned |
| Дата старта | YYYY-MM-DD |
| Дата завершения | YYYY-MM-DD |
| Коммиты | `sha1 short` + сообщение |
| Связанные spec'ы | ссылки на features/NNN |

## Проблема
Что сломано / чего не хватает. Симптомы, контекст, user-impact.

## Диагностика
Как нашли root cause. Какие эксперименты, какие ложные следы.

## Решение
Что реально сделали. С референсом на конкретные файлы/строки.

## Риски и edge cases
Что может пойти не так. Что намеренно **не** покрыто.

## Верификация
Как проверено. Unit / manual / smoke build. Критерии приёмки.

## Нерешённое / follow-up
Откладывается в отдельные задачи (с указанием какие).
```

## Когда заводить

- Любой баг с нетривиальным root cause (не «поправил опечатку»)
- Любая рефакторинг-сессия с архитектурными последствиями
- Любой performance-pass с замерами
- Релиз-подготовка (собрать changelog + проверить migration)

## Когда НЕ заводить

- Тривиальные фиксы (опечатка, один-на-один замена, локальная правка без side effects)
- Fearures — для них есть `features/NNN/spec.md`. Но к feature может прилагаться task «реализация X для feature Y».
